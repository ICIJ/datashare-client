# Storybook `$core` + MSW Mocking Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the failing Storybook stories render without errors and with realistic mocked data, by provisioning a faithful `$core` global and intercepting backend requests with Mock Service Worker.

**Architecture:** Two composing layers. **Layer A** installs a real `Core` (reusing `tests/unit/CoreSetup.js`) onto Storybook's preview app so `useCore()`/`useMode()` stop returning `undefined` — this alone fixes the prop-driven components that make up most of the failing list. **Layer B** adds `msw` + `msw-storybook-addon` so the real `apiInstance` (axios) and `elasticsearch-browser` client get their requests answered from fixtures, with a per-resource handler set and declarative per-story overrides.

**Tech Stack:** Storybook 10 (`@storybook/vue3-vite`), Vue 3, MSW 2 (`msw`, `msw-storybook-addon`), `@storybook/test-runner` for verification, the existing `Core` class and `CoreSetup` test helper.

## Background facts (verified against the codebase)

- Failing components call `useCore()` → `proxy.$core`. `.storybook/preview.js` never installs `$core`, so they throw on mount. (`src/composables/useCore.js`, `src/core/Core.js:217`.)
- `Core`'s constructor already sets `this._auth = new Auth()` (`src/core/Core.js:71`) and `get mode()` derives from config with a `LOCAL` default (`src/mode/index.js:25`). So `core.auth` and `core.mode` work **without** running the async `core.configure()` — exactly what `tests/unit/CoreSetup.js` relies on. **We do not call `configure()`** (avoids boot-time endpoint coupling and deadlocks); instead we seed minimal config.
- `tests/unit/CoreSetup.js` is a `Core` subclass that exposes a `plugins` getter `[plugin, bootstrapVue, i18n, [murmur,…], pinia, vueScrollTo, vCalendar, router]`. Tests install these via `mount(C, { global: { plugins: core.plugins } })`. We do the equivalent: install `core.plugins` onto Storybook's preview `app`.
- The failing components are overwhelmingly **prop-driven** (e.g. `DocumentUserComments`, `DocumentUserTags`, `BatchSearchCard`, `ProjectRow`, `DocumentCard` do not fetch on mount). Layer A fixes these. Layer B covers the few real fetches plus gives the design system a maintainable mocking utility.
- Request URLs are path-based; matching with `*/api/...` works regardless of the GH Pages origin (`icij.github.io`). `Api.getFullUrl` uses `VITE_DS_HOST || window.location` (`src/api/index.js:302`); ES host is `…/api/index/search` (`src/api/elasticsearch.js:682`). Both stay unset in Storybook.
- Response shapes (from `tests/unit/specs/store/modules/*.spec.js`): tags = `[{ label }]`; `getRecommendationsByProject` = `{ aggregates: [{ item: { id }, count }], totalCount }`; `getDocumentsRecommendedBy` = array of doc ids; `getUser` ≈ `{ uid }`.

- [ ] **Step 1: Create the core installer (reusing `CoreSetup`)**

Create `.storybook/decorators/core.js`:
```js
import CoreSetup from '../../tests/unit/CoreSetup'

// Build the Storybook core exactly once and reuse it across stories.
let core = null

export function getStorybookCore() {
  if (core) {
    return core
  }
  core = CoreSetup.init().useAll().useRouterWithoutGuards()
  // Seed the minimal config that prop-driven components read (mode, projects).
  // We intentionally do NOT call `configure()` (no backend round-trip on boot).
  core.config.merge({
    mode: 'LOCAL',
    projects: [{ name: 'local-datashare', label: 'Local Datashare' }],
    defaultProject: 'local-datashare',
    dataDir: '/home/datashare/data'
  })
  // Expose globally so plugin-style components (and the Core API docs) resolve.
  if (typeof window !== 'undefined') {
    window.datashare = core
  }
  return core
}

// Install the core's Vue plugins onto Storybook's preview app, mirroring how
// unit tests pass `core.plugins` to `mount(..., { global: { plugins } })`.
export function installCore(app) {
  const built = getStorybookCore()
  for (const plugin of built.plugins) {
    if (Array.isArray(plugin)) {
      app.use(...plugin)
    } else {
      app.use(plugin)
    }
  }
}
```

```js
import { defineComponent, ref, onMounted } from 'vue'

import { useCore } from '@/composables/useCore'
import { errorJson } from '../../../../.storybook/msw/helpers'

// A tiny component that fetches through the real core API so we can demonstrate
// (and verify) that MSW intercepts requests and that per-story overrides apply.
const MswProbe = defineComponent({
  name: 'MswProbe',
  setup() {
    const core = useCore()
    const tags = ref(null)
    const error = ref(null)
    onMounted(async () => {
      try {
        tags.value = await core.api.getTags('local-datashare', 'doc_01')
      } catch (e) {
        error.value = String(e?.message ?? e)
      }
    })
    return { tags, error }
  },
  template: `
    <div>
      <p v-if="error" data-testid="probe-error">Error: {{ error }}</p>
      <ul v-else-if="tags" data-testid="probe-tags">
        <li v-for="tag in tags" :key="tag.label">{{ tag.label }}</li>
      </ul>
      <p v-else>Loading…</p>
    </div>
  `
})

export default {
  title: 'Components/_Utilities/MswProbe',
  component: MswProbe,
  tags: ['autodocs']
}

// Uses the default `tags` handler → renders tag_01 / tag_02.
export const Default = {}

// Overrides only the tags endpoint to 500 → renders the error branch.
export const Error = {
  parameters: {
    msw: { handlers: [errorJson('*/api/:project/documents/tags/:documentId')] }
  }
}
```

## Self-review notes

- **Spec coverage:** Layer A (`decorators/core.js` + `preview.js`) ↔ spec §"Layer A". MSW worker/init/loader/helpers (Task 4) and per-resource handlers/fixtures (Task 5) ↔ spec §"Layer B", §"Fixture & handler organization". GH Pages static-build check (Task 6) ↔ spec §"GitHub Pages / static build". Error/empty variants ↔ `errorJson`/`emptyJson` + `MswProbe Error` story.
- **Deviation from spec (intentional, lower risk):** the spec described `configure()` firing `getSettings`/`getUser` on boot. The plan instead seeds config and skips `configure()`, matching how `CoreSetup` renders components in the unit tests — this removes the boot-deadlock risk the spec flagged while still providing the same `$core`. The `settings`/`auth` handlers remain in the default set so that any code path that does call them is still answered.
- **Naming consistency:** `getStorybookCore()`/`installCore()`, `defaultHandlers`, `emptyJson()`/`errorJson()` are used identically across all tasks.
- **No placeholders:** every code and fixture step contains complete content.
